# CTDE-SAC 项目设计说明

本文档用于帮助快速看懂当前师兄提供的 `CTDE-SAC` 代码。它不是论文中的正式方法章节，而是一份面向后续修改、答辩解释和复现实验的代码说明书。

## 1. 项目定位

当前项目实现了一个“风光发电 + 储能电池 + 多台电解槽制氢”的强化学习调度原型。代码试图解决的问题是：

在离网场景下，风电和光伏发电功率随天气波动，系统需要把这些不稳定电力分配给储能电池和多台电解槽。调度策略希望在提高制氢效率和制氢量的同时，尽量提高可再生能源消纳率，并减少电池充放电方向频繁切换。

项目名称为 `CTDE-SAC`，意思大致是“集中训练、分散执行的多智能体 SAC”。不过从当前代码实现看，它更接近“多智能体独立 SAC + 共享经验回放池 + 共享全局奖励”。严格意义上的 CTDE 通常会让 critic 输入联合状态和联合动作；当前代码中每个智能体的 Q 网络只使用自己的 14 维状态和 1 维动作。因此答辩时建议表述为“基于多智能体 SAC 的调度框架”，不要过度强调已经实现严格 CTDE。

## 2. 设计场景

### 2.1 系统组成

当前仿真系统包含以下部分：

- 新能源发电侧：
  - 风力发电机组，额定功率 `7000 kW`。
  - 光伏发电系统，额定功率 `5000 kW`。
  - 输入数据为逐小时风速、温度、光强。

- 储能侧：
  - 一组储能电池。
  - 容量 `6000 kWh`。
  - 最大充放电功率 `6000 kW`。
  - SOC 下限 `0.2`，上限 `0.8`，初始 SOC 为 `0.2`。

- 制氢侧：
  - 当前代码设置 4 台电解槽。
  - 单台最大功率 `2500 kW`。
  - 单台最小运行功率 `250 kW`。
  - 低于最小运行功率时按待机功率 `50 kW` 处理。
  - 当前代码隐含地把前两台电解槽作为“线性制氢模型”，后两台作为“非线性制氢模型”。后续可以把它解释并改造为 `2 台碱性电解槽 + 2 台 PEM 电解槽` 的混合阵列。

### 2.2 时间尺度

每个时间步代表 1 小时。每个 episode 默认包含 24 个时间步，也就是一天。

训练模式下，环境会从全年气象数据中随机选取连续 24 小时作为一个 episode。测试模式下，环境默认从测试数据的第 0 小时开始运行 24 小时。

### 2.3 数据文件

数据位于 `data/` 目录：

- `风速-温度-光强-全年.csv`：训练数据，包含全年逐小时气象数据。
- `春典型月15日天气数据.csv`：春季典型日测试数据。
- `夏典型月15日天气数据.csv`：夏季典型日测试数据。
- `秋典型月15日天气数据.csv`：秋季典型日测试数据。
- `冬典型月15日天气数据.csv`：冬季典型日测试数据。

每行数据格式为：

```csv
风速,温度,光强
```

当前 CSV 文件没有标题行，代码直接按逗号分隔每行的 3 个数值。

项目主线代码集中在 5 个文件中：

```text

agent.py        SAC 智能体：策略网络、Q 网络、经验回放、多智能体封装
train.py        训练入口：运行 episode、收集经验、更新网络、保存模型
test.py         测试入口：加载模型、评估性能、导出 Excel
```

其他目录和文件：

```text
data/           训练和测试天气数据
models/         已训练模型权重和训练曲线
test_results/   已有测试结果 Excel
新建文件夹/      旧版或备份代码，暂时不建议作为主线修改

```

## 4. 环境设计：`environment.py`

`EnergySystemEnv` 是整个项目的仿真环境。强化学习中的“环境”负责接收智能体动作，并返回下一状态、奖励、是否结束和辅助信息。

### 4.1 初始化

环境初始化时会根据模式读取数据：

- `mode='train'`：读取 `data/风速-温度-光强-全年.csv`。
- `mode='test'`：读取 `config.py` 中配置的测试数据，当前默认是 `data\冬典型月15日天气数据.csv`。

如果读取失败，代码会使用一组 24 小时的备用常数数据，但正式实验不应该依赖备用数据。

### 4.2 状态空间

每个智能体得到相同的 14 维全局状态。状态包括：

1. 当前小时归一化值。
2. 当前风速相对额定风速的比例。
3. 当前温度归一化值。
4. 当前光强归一化值。
5. 电池 SOC（电池电量）。
6. 电池上一时刻状态，充电为正、放电为负、静止为 0。
7. 当前累计切换次数归一化值。
8. 当前风电功率相对风机额定功率的比例。
9. 当前光伏功率相对光伏额定功率的比例。
10. 当前风光总功率相对理想总功率的比例。
11. 当前风光总功率相对电解槽总容量的比例。
12. 当前理论最大制氢量归一化值。
13. 电池 SOC 剩余可充空间。
14. 一个综合特征，用于表示风光功率富余程度与电池可充空间的组合。

当前所有智能体都拿到同一份 `global_state`：

- `battery`
- `electrolyzer_0`
- `electrolyzer_1`
- `electrolyzer_2`
- `electrolyzer_3`

### 4.3 动作空间

系统共有 5 个动作：

1. 电池动作：由 `battery` 智能体输出。
2. 1 号电解槽动作：由 `electrolyzer_0` 输出。
3. 2 号电解槽动作：由 `electrolyzer_1` 输出。
4. 3 号电解槽动作：由 `electrolyzer_2` 输出。
5. 4 号电解槽动作：由 `electrolyzer_3` 输出。

动作原始范围约为 `[-1, 1]`。环境会把动作缩放到实际功率：

- 电池功率 = 动作值 × `6000 kW`。
- 电解槽功率 = 动作值 × `2500 kW`。

电池功率在当前代码中的符号含义：

- 正值：电池充电，消耗多余可再生能源。
- 负值：电池放电，补充电解槽用电。

电解槽动作的含义：

- 正值且足够大：生产制氢；
- 很小的正值、0、负值：“待机”，消耗 50 kW；最小运行功率250kW

### 4.4 风电出力模型

风电出力由当前风速计算：

- 风速低于切入风速 `1 m/s` 时，输出为 0。
- 风速高于切出风速 `25 m/s` 时，输出为 0。
- 风速低于额定风速 `12 m/s` 时，使用风机功率公式估算。
- 风速高于额定风速且低于切出风速时，输出额定功率 `7000 kW`。

这部分是简化物理模型，不是高精度风机仿真。

### 4.5 光伏出力模型

光伏出力由温度和光强计算。代码考虑了温度偏差和相对辐照度，并最终把输出限制在 `[0, 5000 kW]` 范围内。

这部分同样是简化模型，适合用于强化学习环境中的近似仿真。

### 4.6 电池和电解槽物理约束

环境执行动作时，不会完全照单执行智能体给出的功率，而是会做约束修正：

- 电解槽功率低于最小运行功率 `250 kW` 时，按待机功率 `50 kW` 处理。
- 电解槽功率超过单台最大功率 `2500 kW` 时，截断到最大功率。
- 如果所有电解槽请求功率超过风光发电和电池可放电能力，则按比例削减电解槽功率。
- 电池充放电受到 SOC 上下限和最大功率限制。
- 最终尽量保持功率平衡：风光发电 = 电解槽用电 + 电池充电功率。

### 4.7 制氢量计算

当前代码中，4 台电解槽被分成两类：

- 前两台：线性制氢模型。
- 后两台：二次非线性制氢模型。

对应公式大致为：

```text
线性电解槽制氢量 = linear_coef × (power / 1000) + fixed_consumption
非线性电解槽制氢量 = nonlinear_coef_a × (power / 1000)^2 + nonlinear_coef_b × (power / 1000) + standby_consumption
```

这里的 `fixed_consumption` 和 `standby_consumption` 从命名看像“消耗”，但在代码中是加到制氢量上的常数项。后续如果要严谨化，需要重新检查这些参数的物理含义。

### 4.8 奖励函数

奖励函数由 4 部分组成：

```text
reward = 制氢效率奖励 + 消纳率奖励 - 切换惩罚 - SOC 越界惩罚
```

其中：

- 制氢效率奖励：实际制氢量 / 当前理论最大制氢量，再乘以权重 `1000.0`。
- 消纳率奖励：当前可再生能源消纳率，权重为 `1`。
  -  新能源消纳率 = 被使用的风光电量（电解槽消耗风光发的电+风光向电池充电吸收的电） / 风光总发电量
- 切换惩罚：电池从充电切到放电，或从放电切到充电时产生惩罚，权重为 `2`。
- SOC 越界惩罚：如果 SOC 超出上下限，惩罚 `5.0`。

奖励函数的中心目标明显偏向制氢效率，因为制氢效率权重远大于其他项。

## 5. 智能体设计：`agent.py`

`agent.py` 实现了 SAC 算法和多智能体封装。

### 5.1 ReplayBuffer

`ReplayBuffer` 是经验回放池。它保存：

```text
state（14维）, action, reward, next_state, done
```

训练时，每走一步环境，就把这一条经验放进回放池。网络更新时，从回放池中随机采样 batch，打乱样本相关性，提高训练稳定性。

### 5.2 GaussianPolicy

`GaussianPolicy` 是策略网络，也就是 SAC 中的 actor。

它接收状态，输出动作分布的均值和标准差。随后通过高斯分布采样动作，并使用 `tanh` 把动作压缩到 `[-1, 1]` 附近。

对于电池智能体，代码还加入了一些特殊处理：

- 根据疑似 SOC 信息调整动作均值。
- 训练时额外加入随机噪声，使电池更积极探索充放电行为。

### 5.3 QNetwork

`QNetwork` 是价值网络，也就是 SAC 中的 critic。

它采用双 Q 网络结构：

- `q1`
- `q2`

每个 Q 网络输入为当前智能体的状态和动作，输出一个 Q 值。训练目标中会取两个 Q 值中较小的一个，以缓解 Q 值过估计问题。

### 5.4 SACAgent

`SACAgent` 表示单个 SAC 智能体。每个智能体包含：

- 一个策略网络 `policy`。
- 一个当前 Q 网络 `q_network`。
- 一个目标 Q 网络 `target_q_network`。
- actor 优化器。
- critic 优化器。
- 可选的自动温度参数 `alpha` 调整机制。

更新流程如下：

1. 从经验池采样一批数据。
2. 取出属于当前智能体的状态和动作。
3. 用目标 Q 网络计算 target Q。
4. 更新当前 Q 网络。
5. 根据 `alpha * log_prob - Q` 更新策略网络。
6. 如果启用自动调节，则更新 `alpha`。
7. 对目标 Q 网络做软更新。

### 5.5 CTDESACAgents

`CTDESACAgents` 是多智能体管理器。它会创建：

- 1 个电池智能体：`battery`。
- 4 个电解槽智能体：`electrolyzer_0` 到 `electrolyzer_3`。
- 1 个共享经验回放池。

它主要负责：

- 调用所有智能体选择动作。
- 调用所有智能体更新参数。
- 保存和加载所有模型。

## 6. 训练流程：`train.py`

训练入口是：

```bash
python train.py
```

完整流程：

1. 读取配置。
2. 创建训练环境 `EnergySystemEnv(mode='train')`。
3. 创建多智能体系统 `CTDESACAgents`。
4. 创建按时间戳命名的模型保存目录。
5. 对每个 episode：
   - 环境随机选取一个 24 小时片段。
   - 每个小时，智能体根据当前状态输出动作。
   - 环境执行动作并返回奖励和下一状态。
   - 把经验放入共享回放池。
   - 当回放池样本数超过 batch size 后，更新所有智能体。
6. 记录每个 episode 的奖励、制氢量、效率、消纳率和切换次数。
7. 按不同标准保存模型。

训练总共使用1000个24小时片段（Episode）

训练过程中会保存 4 类模型：

```text
best_model_avg_efficiency     滑动窗口平均制氢效率最好
best_model_single_efficiency  单回合制氢效率最好
best_model_single_hydrogen    单回合制氢量最好
final                         训练结束时的最终模型
```

保存位置类似：

```text
models/20250710-161950/best_model_avg_efficiency/
```

每个模型目录内会包含 10 个权重文件：

```text
battery_policy.pth
battery_q.pth
electrolyzer_0_policy.pth
electrolyzer_0_q.pth
...
electrolyzer_3_policy.pth
electrolyzer_3_q.pth
```

## 7. 测试流程：`test.py`

测试入口示例：

```bash
python test.py --model_path models/最佳模型2.0
```

测试脚本会比较同一模型目录下的 4 类模型：

```text
best_model_avg_efficiency：训练到某一时刻时，最近 50 个有效 episode 的平均制氢效率达到了训练过程中的最高值。
best_model_single_efficiency：current_efficiency =训练过程中，某一个 episode 的制氢效率达到了历史最高。
best_model_single_hydrogen：训练过程中，某一个 episode 的总制氢量达到了历史最高。
final：训练结束最后一刻的模型。
```

每类模型会运行 100 轮测试。每轮测试包含 24 小时。测试会统计：

- 总制氢量。
- 总体制氢效率。
- 平均消纳率。
- 总切换次数。
- 每小时风电功率。
- 每小时光伏功率。
- 每小时电池功率。
- 每小时 4 台电解槽功率。
- 每小时 SOC。

最后，测试脚本会选出全局制氢量最好的轮次，并保存为：

```text
best.xlsx
```

注意：当前 `test.py` 默认只读取 `config.py` 中配置的冬季测试数据。如果要做春、夏、秋、冬四季对比，需要修改测试数据路径，或新增一个批量测试脚本。

## 8. 当前实现与开题设想的对应关系

| 开题/综述中的设想 | 当前代码是否覆盖 | 说明 |
| --- | --- | --- |
| 风光制氢系统建模 | 部分覆盖 | 已有风电、光伏、电池、电解槽简化模型 |
| 碱性电解槽与 PEM 电解槽混合阵列 | 部分覆盖 | 当前有两类制氢函数，但没有显式命名为碱性/PEM |
| 电解槽启停/待机状态 | 部分覆盖 | 低于最小功率时进入待机功率，但没有完整三状态模型 |
| 单台电解槽设备级调度 | 覆盖 | 每台电解槽对应一个智能体 |
| SAC 强化学习调度 | 覆盖 | 已实现多智能体 SAC |
| 严格 CTDE 联合训练 | 不完全覆盖 | 当前 critic 没有使用联合状态/联合动作 |
| 多时间尺度优化 | 未覆盖 | 当前为小时级单时间尺度 |
| MILP 对比 | 未覆盖 | 可用规则基线对比替代，作为短期稳妥方案 |
| 实物实验验证 | 未覆盖 | 建议放入不足与展望 |

其中已有图里的 `Consumption Rate` 坐标显示异常，后续建议重新生成或单独绘制更适合答辩展示的消纳率图。

## 10. 当前代码的主要风险点

### 10.1 CTDE 表述风险

当前实现不是严格 CTDE。建议在文档和答辩中使用：

```text
多智能体 SAC 调度框架
```

而不是反复强调：

```text
严格 CTDE-SAC 算法
```

### 10.2 电解槽类型不够明确

当前代码只有“线性”和“非线性”制氢函数，没有把它们显式命名为碱性电解槽和 PEM 电解槽。后续最值得做的小改进之一，就是在配置中加入电解槽类型字段，并在代码和图表中明确展示。

### 10.3 电池 SOC 单位可能需要复核

SOC 更新中使用了：

```text
soc_change = battery_power / (battery_capacity * battery_voltage)
```

但 `battery_capacity` 已经是 `kWh`，再乘以电压从单位上看不一定合理。短期可以把它作为师兄原型中的简化设定保留，但如果后续重构，建议重新检查电池模型。

### 10.4 测试策略仍带随机性

`select_action(evaluate=True)` 时，代码仍然加入了一定噪声。因此测试 100 轮的结果不是完全确定策略结果，而是带随机扰动的评估结果。这可以解释为“鲁棒性测试”，但如果要做严格复现实验，需要去掉评估噪声或固定随机种子。

### 10.5 制氢效率可能超过 1

已有训练曲线中制氢效率出现超过 1 的情况。这说明理论最大制氢量的计算和实际制氢量计算可能不完全一致，或者常数项处理导致实际值高于理论值。答辩展示前建议重新检查指标定义，必要时把图表改成“相对制氢表现”或修正理论最大值计算。

## 11. 后续最应该做的工作

按照“稳过好看 + 可控加分”的目标，建议优先级如下：

1. 先读懂 `environment.py`，尤其是动作如何被修正、制氢量如何计算、奖励如何给出。
2. 再读懂 `agent.py`，重点理解 SAC 的 actor、critic、replay buffer 和多智能体封装。
3. 然后跑通已有模型测试，拿到 `best.xlsx`，确认结果表中的每一列含义。
4. 增加四季典型日测试和图表输出，把结果整理成答辩可用图片。
5. 最后再做小改进：显式混合电解槽类型、规则基线对比、README 修复。

短期不建议优先做大规模算法重构，因为训练稳定性和调试成本不可控。当前最有价值的路线是：把已有代码讲清楚、跑稳定、图表做漂亮，并用文献综述中的问题意识解释为什么这样设计。
